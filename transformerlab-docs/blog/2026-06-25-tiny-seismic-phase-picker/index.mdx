_We built a very small neural network, a program that learns from examples, that reads earthquake recordings and marks when the shaking starts. It does that job as well as the big standard tools._

> **Quick summary:** When an earthquake happens, a sensor records two bumps: a small early one (the P wave) and a bigger later one (the S wave). Software has to mark exactly when each one arrives. The popular tools for this are large neural networks. We trained a tiny one instead, with about 34,000 parameters (the values a network learns). We tested it carefully so it could not peek at the answers. It scored 0.76 out of a possible 1.0, where higher is better. A standard model called PhaseNet scored 0.64, and it is about 8 times bigger. One simple change to how we trained our model was the key. We also tested self-supervised pretraining, a popular method where a model first studies lots of unlabeled recordings before training on the few labeled recordings. It did not actually help here, and one lucky test run made it look like it did.

## What the model has to do

Picture a seismometer, a machine that records ground shaking. When an earthquake hits, two kinds of waves reach it a few seconds apart. The P wave gets there first. It is a small, quick jolt. The S wave comes next, and it is the bigger shake you would feel.

The exact moment each wave arrives is the single most useful number in earthquake monitoring. From those arrival times, scientists figure out where the quake was, how big it was, and whether to send an alert. Marking those two moments on a wiggly recording is called "phase picking."

People used to do this with hand-written rules. Now they use neural networks, which are programs that learn from examples. Two of them, PhaseNet and EQTransformer, are the go-to tools, and they work well. But two habits in how they are built and graded are worth questioning.

First, they are big. A network's size is counted in "parameters," the numbers it adjusts while learning. The standard pickers have hundreds of thousands of them. Second, they are usually graded on a test that is too easy by accident. One earthquake is recorded by many sensors, and each recording gets chopped into overlapping clips. If you shuffle all those clips and split them at random into a practice pile and a test pile, near-copies of the same quake end up in both piles. The model has already seen the test answers during practice. That is called data leakage, and it makes the scores look better than they really are.

We wanted to drop both habits at once and ask a simple question. How small can a picker be, and how good is it when the test is honest?

## A fair test, and a small model

We used a public collection called STEAD: about 1.2 million earthquake recordings, each one already marked by a human expert with the true arrival times. We split it so that every earthquake goes entirely into either the practice set or the test set, never both. That way the test has no sneaky copies. We only count a pick as correct if it lands within a tenth of a second of the true P arrival, or two tenths of a second for the slower S wave. The score we report throughout is an F1 score on a 0-to-1 scale, where 1 is perfect. It rewards the model for marking each arrival at the right instant: catching the real P and S onsets while not crying wolf on quiet stretches.

Our model is a small network with 33,610 parameters. That is tiny by today's standards, small enough to run on an ordinary computer without special hardware.

The first try failed in a strange way. The model learned to find the S wave but completely ignored the P wave. Its P score was 0.00, a total miss. Here is why. In any recording, almost every moment is just background noise, more than 99 out of 100. When the model trains, it earns the most reward for labeling all that quiet correctly, so it labels everything as noise. It never learns to flag the faint little P bump.

The fix is to change the scoring during training. We reward the model far more for catching the rare P and S moments than for labeling the common quiet. That one change pulled the score on our tuning checks from 0.40 up to 0.78, and the P score from 0.00 up to 0.78. The S wave, which is louder and easier, was already fine. So careful training, not a bigger network, is what made the small model work.

## The result: small beats big

On the test recordings it had never seen before, our tiny model scored 0.76 (0.78 for P, 0.75 for S). We then ran the standard PhaseNet through the exact same test, the same split and the same grading, so it is a fair, side-by-side match. PhaseNet scored 0.64, with about 270,000 parameters, roughly 8 times more than ours.

So the smaller model is the more accurate one here. Two honest notes. We tested PhaseNet using our own setup rather than retraining it on our split, so its score may differ from what a fresh training would give. And our own number comes from a single training run. The paper keeps both notes. Even so, the point stands. A picker small enough to run almost anywhere is not giving up accuracy to be that small.

## The idea that did not work

When you only have a few labeled examples, which is the real situation when you set up monitoring somewhere new, there is a popular method called self-supervised pretraining. The model first studies a big pile of recordings with no answers attached, just to learn what earthquake signals look like. That is the "self-supervised" part: it learns from the raw data itself, with no labels. Then you fine-tune it (train it a little more) on your small set of labeled examples. The hope is a head start.

We tried it. With only 1 in 100 labels (about 100 examples), the pretrained version scored 0.15 higher. That looked like a clear gain.

Then we re-ran the test several more times, each with a different random starting point, called a "seed." Because the computer makes some random choices while it learns, the result wobbles a bit each time you run it. With only 100 examples, a single run can land lucky or unlucky. Across three seeds the gains were +0.008, +0.078, and -0.125, two tiny gains and one loss. Add them up and the average is basically zero. The first +0.15 was just one lucky run. It did not help at 5 in 100 labels either.

<SeedTrapDiagram />

The likely reason is that our carefully trained small model already does well with few labels, so there is nothing left for the pretraining to add. The bigger lesson is about honest testing. One run can show a confident gain that is pure luck. You only learn the truth by repeating it several times.

## Where it does not work

The model has real limits. How well it does depends mostly on how far away the earthquake is. For nearby quakes it scores 0.93. For quakes more than 150 kilometers away, where the signal is faint, it drops to 0.38 (measured on only 54 far-away quakes, so that number is rough). It also raises false alarms on quiet data: when there is no earthquake at all, it still calls a P pick 69 percent of the time and an S pick 51 percent of the time. In a real system you would put a separate earthquake-detection step in front of it to catch those false alarms. Our score measures picking inside a clip that already has a quake in it, not finding the quake from scratch.

## Takeaways

- A 34,000-parameter model matches a much bigger standard picker on a fair test. For this job, accuracy is not about size.
- The key to a small picker is rewarding it for the rare P and S moments. Without that, it never learns to find the P wave.
- Self-supervised pretraining (study first, then fine-tune) gave no real benefit here, and a single test run almost faked one. Always repeat low-label tests with several seeds.
- The model gets worse for far-away quakes and raises false alarms on quiet data, so it should run behind a separate earthquake detector, not alone.
