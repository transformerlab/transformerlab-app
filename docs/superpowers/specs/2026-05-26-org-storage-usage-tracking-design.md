# Org Storage Usage Tracking & Limits — Design

**Date:** 2026-05-26
**Status:** Approved (pending spec review)

## Goal

Give org/team admins visibility into how much disk space their org consumes
(workspace files + local provider runs), surface it in the Usage Report area,
let them set notify thresholds, and enforce a deployment-wide per-org hard cap
that blocks job launches when an org's storage exceeds it.

## Summary of Decisions

- **Measurement strategy:** background scan + cached DB snapshots (not lazy/on-demand).
- **Limit model:**
  - **Global per-org cap** (uniform, set by deploy via env var / global config) is
    the only **hard-blocking** limit. Each org is measured independently against it:
    if Org A is over the cap its launches are blocked until it drops back under,
    while Org B (under the cap) is unaffected.
  - **Org-level** and **user-level** thresholds are **notify-only**.
- **Notifications:** reuse the existing webhook `notification_service` machinery
  **and** show an in-app banner. No email (no existing email infra).
- **Global cap configuration:** env var / global `Config` key only (no UI; managed by
  whoever deploys the instance).
- **v1 scoping:** a single per-team user notify threshold (not true per-user values);
  enforcement **fails open** when no snapshot exists yet.

## Section 1 — Data Model & Measurement

### New table `OrgStorageSnapshot`

Follows the `QuotaUsage` pattern. **No foreign keys** (per repo rules). Naive UTC
datetimes via `utc_now_naive()`.

| Column            | Type      | Notes                                                        |
|-------------------|-----------|--------------------------------------------------------------|
| `id`              | int PK    |                                                              |
| `team_id`         | str       | indexed                                                      |
| `total_bytes`     | int       | total org usage (workspace + local provider runs)           |
| `breakdown_json`  | text      | bytes per category (see below)                               |
| `per_user_json`   | text      | best-effort bytes attributed per `user_id`                   |
| `scanned_at`      | datetime  | naive UTC                                                    |

Categories in `breakdown_json`: `workspace_models`, `workspace_datasets`,
`workspace_experiments`, `workspace_other`, `local_provider_runs`.

UI and enforcement always read the **most recent row per `team_id`**. History rows
are retained (cheap, enables trend charts later); no pruning in v1.

Alembic migration autogenerated where possible; no FKs.

### SDK addition — `storage.du`

Add to `lab-sdk/src/lab/storage.py`:

```python
async def du(path: str) -> int:
    """Total bytes under path across local + remote (S3/GCS/Azure) backends.
    Returns 0 if the path does not exist."""
    if not await exists(path):
        return 0
    fs = await filesystem()
    try:
        return await asyncio.to_thread(fs.du, path, total=True)
    except (NotImplementedError, TypeError):
        # Fallback: sum file sizes from walk()/info() for backends lacking du.
        total = 0
        for root, _dirs, files in await walk(path):
            for f in files:
                info = await asyncio.to_thread(fs.info, join(root, f))
                total += int(info.get("size", 0) or 0)
        return total
    finally:
        await _close_filesystem(fs)
```

`fsspec`'s `fs.du(path, total=True)` is supported by the local, S3, GCS, and Azure
backends, giving correct sizing for both local and remote workspaces.

> **SDK reinstall gotcha:** after changing `lab-sdk/`, run `cd lab-sdk && pip install -e .`
> and restart the API — the API imports the *installed* `lab` package.

### Measurement service — `storage_usage_service.py`

`api/transformerlab/services/compute_provider/storage_usage_service.py`:

- `compute_org_storage(team_id) -> dict`: `set_organization_id(team_id)`, then
  `storage.du()` over `get_workspace_dir()` subdirs (models/datasets/experiments/other)
  and `get_local_provider_org_dir(team_id)`. Returns total + breakdown + best-effort
  per-user map.
- Per-user attribution is **best-effort**: experiment/job dirs whose `job_data` carries
  `user_info` are attributed to that user; unattributable bytes count only toward the org total.
- `write_snapshot(session, team_id, result)`: inserts one `OrgStorageSnapshot` row.
- `get_latest_snapshot(session, team_id)`: most recent row for an org.

## Section 2 — Background Scan Worker

Modeled on `notification_service._notification_worker_loop`, started at app startup.

- Interval via env var `TFL_STORAGE_SCAN_INTERVAL_SECONDS` (default 900s / 15 min).
- Each tick: iterate all teams → `compute_org_storage` → `write_snapshot` →
  evaluate thresholds (Section 4).
- Per-org `try/except`: one org's failure logs and is skipped, never kills the loop.
- Manual trigger: `POST /compute_provider/usage/storage/rescan` (owner-only) forces a
  fresh scan for the caller's org without waiting for the next tick.

## Section 3 — Limits, Config & Enforcement

### Config keys (existing `Config` store)

| Key                                  | Scope            | Set by         | Effect        |
|--------------------------------------|------------------|----------------|---------------|
| `storage_global_per_org_limit_bytes` | global (team_id=None) | env var / deploy | **hard cap** |
| `storage_org_notify_threshold_bytes` | per-team         | owner (UI)     | notify-only   |
| `storage_user_notify_threshold_bytes`| per-team         | owner (UI)     | notify-only   |

The global cap is read from env var on startup and written to the global `Config` key
(env var wins if both present).

### Enforcement

In `launch_template.py`, immediately after the existing quota check and before
`job_service.job_create`:

```python
limit = await storage_service.get_global_per_org_limit_bytes(session)
if limit:
    snap = await storage_service.get_latest_snapshot(session, team_id)
    if snap and snap.total_bytes >= limit:
        raise HTTPException(
            status_code=403,
            detail=f"Org storage ({gb(snap.total_bytes)} GB) exceeds the global "
                   f"limit ({gb(limit)} GB). Free space before launching new jobs.",
        )
    # No snapshot yet -> fail open (allow launch), log a warning.
```

Only the global per-org cap blocks. Org/user thresholds never block.

## Section 4 — Notifications & Threshold Evaluation

After each scan, per org compare latest `total_bytes` against thresholds:

- **Org threshold crossed** → fire webhook via `notification_service`; record a
  "notified" flag so repeated ticks don't spam. Re-arm when usage drops below.
- **User threshold crossed** → same, per user.
- **In-app banner** → `GET /compute_provider/usage/storage/alerts` returns active
  alerts for the current org (over-global, over-org-threshold, over-user-threshold);
  frontend renders a warning banner.

Extend `notification_service` with a storage-alert payload type rather than parallel infra.

## Section 5 — Frontend (Storage tab)

- Add a **"Storage" tab** to `Team.tsx` settings tabs (alongside Quota Settings), owner-only.
- Content:
  - Total usage vs. global limit progress bar (reuse `QuotaReportSection` visual style).
  - Category breakdown (table/bar).
  - Per-user breakdown table.
  - "Last scanned" timestamp + **Recalculate now** button (calls rescan endpoint).
  - Inputs for org/user notify thresholds (calls thresholds endpoint).
  - Active-alert banner at top when over a threshold.
- New `useAPI` hooks; TypeScript interfaces for snapshot/alerts responses (no `any`).

### New endpoints (under `compute_provider/usage` router, owner-only)

| Method & path                                       | Purpose                                   |
|-----------------------------------------------------|-------------------------------------------|
| `GET /compute_provider/usage/storage`               | latest snapshot + effective limits        |
| `POST /compute_provider/usage/storage/rescan`       | force a scan for the caller's org         |
| `GET /compute_provider/usage/storage/alerts`        | active alerts for the current org         |
| `PATCH /compute_provider/usage/storage/thresholds`  | set org/user notify thresholds            |

Pydantic schemas in `schemas/`; business logic in the service layer, not routers.

## Out of Scope (v1)

- True per-user threshold values (only a per-team user threshold).
- Email notifications.
- A UI for the global per-org cap (env/config only).
- Snapshot history pruning / retention policy.
- Physical disk free-space guard (only summed per-org usage is enforced).
